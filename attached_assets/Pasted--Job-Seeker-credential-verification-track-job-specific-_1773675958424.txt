## Job Seeker credential verification: track + job-specific, append to active request)

**Goal:** Implement Job Seeker credential/license verification using the same architecture as Employer verification, with:

- **Baseline requirements** based on the seeker’s selected **Track**
- **Additional requirements** based on the **Job context** (job + jurisdiction/state + tags)
- **One active request per seeker**
- If a seeker applies to another job while a request is active, **append new requirements to the existing request** instead of creating a second request.

### A) Database tables (Drizzle in `schema.ts`)

If you already created employer verification tables, mirror them for seekers.

#### 1) `jobSeekers` (if you don’t already have a seeker profile table)

If seekers are currently just rows in `users`, then reuse `users` and treat `userId` as `jobSeekerId`. Otherwise create:

- `id` uuid PK
- `userId` uuid FK → [users.id](http://users.id) (optional depending on your auth model)
- `displayName` text
- `track` text default `"Unknown"` (Driver, Warehouse, Dispatch, Mechanic, Maritime, Aviation, Mining, Other, Unknown)
- `verificationStatus` text default `"Unverified"`
- `notionSeekerUrl` text nullable (future)
- timestamps

#### 2) `seekerVerificationRequests`

- `id` uuid PK
- `jobSeekerId` FK
- `status` text: Draft | Submitted | Needs more | Verified | Rejected
- `submittedAt` timestamp nullable
- `reviewedAt` timestamp nullable
- `trackAtRequestTime` text
- `primaryJobId` uuid nullable (the job that initiated the request)
- `jurisdictionState` text nullable (2-letter)
- `seekerNotes` text nullable
- `reviewerNotes` text nullable
- `requiredRequirementKeysSnapshot` jsonb (array of strings) **required**
- timestamps

Add an index `(jobSeekerId, status)` and enforce one-active-request in code (partial unique if available).

#### 3) `seekerCredentialEvidenceItems`

- `id` uuid PK
- `jobSeekerId` FK
- `requestId` FK → [seekerVerificationRequests.id](http://seekerVerificationRequests.id)
- `requirementKey` text
- `sourceType` text
- `sourceUrl` text nullable
- `documentR2Key` text nullable
- `documentOriginalName` text nullable
- `documentMimeType` text nullable
- `excerpt` text nullable
- `notes` text nullable
- `retrievedAt` date/timestamp nullable
- `decision` text default "To review"
- `confidence` text nullable
- timestamps

#### 4) `seekerCredentialRequirements` (requirements registry in Postgres for runtime)

Store requirements in Postgres so the app does not depend on Notion at runtime.

Columns:

- `key` text PK (snake_case)
- `name` text
- `scope` text: Global | Track | JobRule
- `track` text nullable (Driver/Warehouse/etc)
- `minConfidence` text nullable
- `notes` text nullable
- `isActive` boolean default true

#### 5) `seekerRequirementRules` (job/jurisdiction rules)

Columns:

- `id` uuid PK
- `name` text
- `isActive` boolean
- `appliesToTrack` text nullable
- `appliesToState` text nullable
- `appliesToEmployerCategory` text nullable
- `appliesToJobTag` text nullable
- `requiresRequirementKey` text FK → seekerCredentialRequirements.key

This is the engine for job-specific add-ons.

### B) Core server routes (Express)

Create `server/routes/seekerVerification.ts` mounted at `/api/seeker/verification`.

#### 1) POST `/request/get-or-create`

Input: optional `{ jobId }`

- Identify job seeker from session.
- Find active request where status in Draft/Submitted/Needs more.
- If exists, return it.
- Else create Draft request with:
    - trackAtRequestTime = seeker.track (or Unknown)
    - primaryJobId = jobId or null
    - requiredRequirementKeysSnapshot = computed requirements (see below)

#### 2) GET `/requirements`

Input: `{ jobId? }`

Return:

- baseline requirements from track
- plus job-specific requirements resolved from `seekerRequirementRules` using:
    - seeker.track
    - job.state (jurisdiction)
    - employer.employerCategory
    - job.tags

Return a merged list and the computed set of required requirement keys.

#### 3) POST `/request/append-requirements`

Input: `{ requestId, jobId }`

**This implements your “append to active request” decision.**

- Load active request (must belong to seeker).
- Compute new required keys for the given job.
- Union them with `requiredRequirementKeysSnapshot`.
- Save updated snapshot.
- Do not change status.

#### 4) POST `/evidence`

Input: `{ requestId, requirementKey, sourceType, sourceUrl?, excerpt?, notes?, retrievedAt? }`

- Validate request belongs to seeker and status is Draft or Needs more (allow Submitted only if you explicitly choose, but default disallow).
- Create seekerCredentialEvidenceItems row.

#### 5) POST `/evidence/upload`

If you want uploads now:

- Accept multipart upload, store to Cloudflare R2, write `documentR2Key` etc.
- Return evidence item id.

#### 6) POST `/request/submit`

- Validate completeness: for every `requiredRequirementKeysSnapshot` there is at least one evidence item attached with that requirementKey.
- Set status Submitted + submittedAt.

### C) Apply flow integration (job-specific add-ons)

When a seeker clicks “Apply” on a job:

1. Call `POST /api/seeker/verification/request/get-or-create` with `{ jobId }`
2. Call `POST /api/seeker/verification/request/append-requirements` with `{ requestId, jobId }`
3. If request status is not Verified:
    - Show modal: “Verification required to apply” with checklist and upload UI.
4. Once verified, allow apply.

### D) Admin routes

Under `/api/admin/seeker-verification`:

- `GET /inbox` (Submitted, Needs more)
- `POST /request/decision` (Needs more / Verified / Rejected)
    - If Verified, set seeker.verificationStatus = Verified.

### E) Seed initial requirements + rules (minimal)

Seed `seekerCredentialRequirements` with common keys:

- `cdl_a` (Track=Driver)
- `cdl_b` (Track=Driver)
- `hazmat_endorsement` (Track=Driver or rules by job tag hazmat)
- `twic` (JobRule: appliesToJobTag="port_access" or appliesToTrack="Maritime")
- `forklift_cert` (Track=Warehouse)
- `icao_iata_training` (Track=Aviation) (or rule-based)
- `msha_part48` (Track=Mining or job tag mining_site)

Seed `seekerRequirementRules` for examples:

- if jobTag = "hazmat" requires `hazmat_endorsement`
- if employerCategory = "Marine" requires `twic`
- if track = "Mining" requires `msha_part48`

### One important UX rule to implement

If a seeker already has an active request and applies to a new job, do **not** create a second request. Always:

- append requirements to the existing request snapshot
- show the delta as “New requirements added for this job”
